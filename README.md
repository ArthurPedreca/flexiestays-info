# The Bournemouth Guide

An SEO-first, static, content-driven guide to **Bournemouth, Dorset (UK)** modelled
on the structure of [bournemouth.co.uk](https://www.bournemouth.co.uk/) but with an
original design, original wording and a data model built so that **content pages can be
generated by automation** (e.g. an AI pipeline that rewrites source material into new pages).

Built with **[Astro](https://astro.build)** → outputs 100% static HTML, deploys on Vercel.

---

## Why Astro (and what "the pattern" means here)

The previous project was a single hand-tuned, SEO-heavy HTML file. That pattern static
output, heavy structured data, `llms.txt`/`robots.txt`/sitemap is preserved. What changed:
the design now lives in **one place** (shared layouts + `global.css`), and each page is
generated from a **Markdown file** through a **template**. So the automation writes Markdown,
not raw HTML, and every page inherits the same design and SEO automatically.

## Project structure

```
src/
  content.config.ts        # THE DATA MODEL  Zod schemas for every content type
  content/                 # ← the automation writes Markdown here
    listings/{slug}.md      →  /listing/{slug}        (attractions, beaches, hotels, food…)
    events/{slug}.md        →  /event/{slug}          (dated)
    blog/{slug}.md          →  /blog/{slug}
    areas/{slug}.md         →  /explore/towns|villages/{slug}
    itineraries/{slug}.md   →  /ideas-and-inspiration/itineraries/{slug}
    hubs/*.md               →  editorial intro for category hub pages
    pages/**/*.md           →  generic info/editorial pages (path mirrors URL)
  layouts/                 # one template per page type (Listing, Event, Blog, Area, Hub, Itinerary, Info)
  components/              # Header, Footer, BaseHead (SEO+JSON-LD), Breadcrumbs, Map, Cards, Faq…
  lib/                     # site config, URL builders, JSON-LD builders, date helpers
  pages/                  # routes (mostly thin  they load content + pick a layout)
  styles/global.css       # the whole design system
public/
  images/                 # photos (referenced as /images/…)
  robots.txt · llms.txt · favicon.svg
docs/CONTENT-MODEL.md     # ← human-readable authoring contract (give this to the automation)
legacy/                   # the old reviewsflexiestays.com site, preserved
scripts/new-content.mjs   # scaffolds a new content file with valid frontmatter
```

## Commands

```bash
npm install       # once
npm run dev       # local dev server (http://localhost:4321)
npm run build     # static build → dist/
npm run preview   # preview the built site

# scaffold a new page stub (created as draft: true)
npm run new:listing -- oceanarium-bournemouth "Oceanarium Bournemouth"
npm run new:blog -- best-walks-in-bournemouth
npm run new:event -- bournemouth-air-festival
node scripts/new-content.mjs area southbourne "Southbourne"
```

## How SEO is handled (every page, automatically)

- **Canonical URL, Open Graph & Twitter tags** `components/BaseHead.astro`
- **JSON-LD structured data** `lib/schema.ts` emits the right schema.org type per page
  (`TouristAttraction` / `LodgingBusiness` / `Restaurant` / `Beach` / `Event` /
  `BlogPosting`), plus `BreadcrumbList`, `FAQPage`, `WebSite` + `Organization`.
- **Breadcrumbs** on every interior page (visual + structured).
- **FAQ blocks** with `FAQPage` schema a differentiator vs the source site.
- **Sitemap** generated at build (`/sitemap-index.xml`) by `@astrojs/sitemap`.
- **`llms.txt` + AI-crawler-friendly `robots.txt`** for GEO (AI answer engines).
- **RSS feed** for the blog at `/rss.xml`.

## The automation workflow (the point of this project)

1. Scrape / read a source page about Bournemouth.
2. Rewrite it (design a prompt that outputs **frontmatter + Markdown body** conforming to
   `docs/CONTENT-MODEL.md`). Keep wording original.
3. Write the file to the right `src/content/<collection>/<slug>.md`.
   (Or call `scripts/new-content.mjs` to create a valid stub first.)
4. Commit & push → Vercel rebuilds → the new page is live with full SEO + structured data.

Because the schema is validated at build time (Zod), a malformed generation **fails the build
loudly** instead of publishing a broken page. Cross-links (`related`, `relatedListings`,
`venue`) reference other entries by slug and are resolved automatically.

> **Editorial guardrail:** reword source material do not copy text verbatim. Only reference
> images that exist in `public/images/`; otherwise omit `heroImage` (a branded placeholder shows).

## Deploying to Vercel

This repo is an Astro app. In the Vercel project settings, set the **Framework Preset** to
**Astro** (Vercel auto-detects it; build command `astro build`, output directory `dist`).
Set the production domain to `thebournemouthguide.com` and update `SITE_URL` in
`src/lib/site.ts` and `site` in `astro.config.mjs` if the domain changes.

The old single-page site is preserved untouched in `legacy/` for reference.
